# 3D-Placer: Application Architecture Design

This document outlines the architecture for the 3D-Placer application, an Ionic/Angular-based mobile tool for quick 3D scene prototyping.

## 1. High-Level Vision & Goals

**Prima-   **API Integration: `AssetService`**

-   The `AssetService` will be responsible for generating URLs for the placeholder image API (`https://picsum.photos/`).
-   **Usage Example:** In the `EditorPage`, there could be a button "Set Background".
    -   This button's click handler would call `AssetService.getImageUrl(1024, 768)`.
    -   The service returns the formatted URL string.
    -   This URL is then set as the `backgroundImageUrl` property in the current `Scene` object.
    -   `ThreeJsService` would then be instructed to update the scene's background using the new image URL, loading it as a texture.
-   This provides a simple, clear use case for integrating an external API as requested.* To provide game developers and 3D artists with a mobile-first tool to quickly assemble and visualize 3D scenes using a library of pre-existing models. This is not a full-fledged 3D modeling tool, but a "placer" or "arranger" for quick mockups on the go.

**Core User Stories:**
- As a user, I want to manage a list of different 3D scenes.
- As a user, I want to import 3D models (e.g., GLTF/GLB, OBJ, FBX, DAE) from my device's local storage into the app.
- As a user, I want to create a new scene, add models from my local library to it, and position, rotate, and scale them.
- As a user, I want to save my scenes on my device and be able to load them later.
- As a user, I want to use images from an external API as textures or backgrounds for my scene.

## 2. Core Technologies

- **UI Framework:** Ionic with Angular. The existing side-menu structure will be used for navigation.
- **3D Rendering:** Three.js for all 3D rendering and scene manipulation.
- **Native Functionality:** Capacitor to enable native device features, specifically for file system access to store and retrieve models and scene data.
- **Language:** TypeScript.

## 3. Architectural Principles

- **Service-Oriented Architecture:** Angular services will manage the application's state, separating concerns like 3D rendering, data management, and API communication from the UI components.
- **Component-Based UI:** The UI will be built from modular and reusable Angular components (e.g., a viewport component, a properties editor, a model library view).
- **Decoupled 3D Engine:** The entire Three.js logic will be encapsulated within a dedicated service (`ThreeJsService`). Angular components will interact with this service through a well-defined API, without directly touching Three.js code. This improves maintainability and testability.
- **Offline-First:** The application is designed to work entirely offline once models are downloaded. Scene and model data will be stored locally on the device's file system.

## 5. Component & Service Breakdown

### Services (in `src/app/core/services/`)

-   **`StorageService`**:
    -   **Purpose:** Abstract all interactions with the device's file system using the Capacitor Filesystem API.
    -   **Methods:**
        -   `saveJSON(path: string, data: any): Promise<void>`
        -   `readJSON(path: string): Promise<any>`
        -   `copyFile(sourceUri: string, destinationPath: string): Promise<void>` // Copies a file to the app's data directory
        -   `fileExists(path: string): Promise<boolean>`
        -   `createDir(path: string): Promise<void>`

-   **`SceneManagerService`**:
    -   **Purpose:** Manage the lifecycle of scenes (create, save, load, delete).
    -   **Methods:**
        -   `getScenes(): Promise<Scene[]>` - Reads scene metadata from a central manifest file.
        -   `getScene(id: string): Promise<Scene>` - Loads a specific scene's data from its file.
        -   `saveScene(scene: Scene): Promise<void>` - Saves scene data to a JSON file.
        -   `createNewScene(name: string): Promise<Scene>` - Creates a new scene object and saves it.

-   **`ModelManagerService`**:
    -   **Purpose:** Handle the importing, storage, and retrieval of 3D models from the device.
    -   **Methods:**
        -   `getModels(): Promise<ModelInfo[]>` - Gets a list of all locally stored models.
        -   `importModel(sourceUri: string, name: string): Promise<void>` - Copies a model from a device URI, saves it to the app's internal storage, and adds it to a manifest. Note: This service must handle multi-file formats like OBJ (with .mtl and textures) by identifying and copying all related files.
        -   `getModelPath(modelId: string): Promise<string>` - Returns the local file path for a given model.

-   **`ThreeJsService`**:
    -   **Purpose:** The heart of the 3D editor. It encapsulates all Three.js logic. It does NOT know about Angular components.
    -   **Responsibilities:**
        -   Initialize the Three.js `Scene`, `Camera`, and `WebGLRenderer`.
        -   Attach the renderer's canvas to a DOM element.
        -   Load models (GLTF/GLB) into the scene using the `GLTFLoader`.
        -   Handle user interactions within the 3D view (object selection via raycasting, orbit controls).
        -   Provide methods to manipulate objects in the scene (translate, rotate, scale).
        -   Expose an API for components to call, e.g., `addObject(modelPath)`, `setSelectedObjectPosition(x, y, z)`.
        -   Emit events when the scene changes (e.g., object selected, object transformed) using RxJS Subjects.

-   **`AssetService`**:
    -   **Purpose:** Interact with the external image API.
    -   **API Documentation:**
        -   URL: `https://picsum.photos/`
        -   Usage: Append desired image size (width & height) to the URL (e.g., `https://picsum.photos/800/600`). For a square image, just add one dimension (e.g., `https://picsum.photos/400`).
    -   **Methods:**
        -   `getImageUrl(width: number, height: number): string` - Constructs a URL for a random image of a specific size.

### Pages & Components

-   **`SceneListPage`**:
    -   Displays a list of scenes fetched from `SceneManagerService`.
    -   Allows creating a new scene (which navigates to the editor page with a new ID).
    -   Allows deleting scenes.
    -   Each item in the list links to `editor/[sceneId]`.

-   **`EditorPage`**:
    -   The main workspace for a single scene.
    -   On initialization, it gets the scene ID from the route and tells `SceneManagerService` to load the scene data.
    -   It then instructs `ThreeJsService` to build the scene based on this data.
    -   Contains the `<app-three-viewport>`, `<app-properties-panel>`, and a button to add models from the library.
    -   Listens for events from `ThreeJsService` (e.g., an object being selected) to update the UI (e.g., showing the properties panel).
    -   Has a "Save" button that calls `SceneManagerService.saveScene()`.

-   **`ModelLibraryPage`**:
    -   Shows a grid of downloaded models from `ModelManagerService`.
    -   Includes the `<app-model-importer>` component to add new models.

-   **`ThreeViewportComponent`**:
    -   A simple component that has a `<div>` or `<canvas>` element.
    -   On `ngAfterViewInit`, it passes its native element to `ThreeJsService` to be used as the rendering target.

-   **`PropertiesPanelComponent`**:
    -   Subscribes to `ThreeJsService`'s `selectedObject$` observable.
    -   When an object is selected, it displays input fields for its position, rotation, and scale.
    -   When the user changes a value in an input, it calls the corresponding method in `ThreeJsService` (e.g., `setSelectedObjectPosition(...)`).

## 6. Data Flow & State Management

-   **State Management:** For this application's complexity, we will rely on Angular's RxJS-based services for state management. We will use `BehaviorSubject` within services to hold and stream state.
    -   `SceneManagerService` will have a `scenes$` observable.
    -   `ModelManagerService` will have a `models$` observable.
    -   `ThreeJsService` will have a `selectedObject$` observable.
-   **Data Persistence:**
    -   All data will be stored on the device's local file system using the Capacitor Filesystem API, managed by our `StorageService`.
    -   A single `manifest.json` file in the app's data directory will store lists of all scenes and models.
    -   Each scene will be saved as a separate `scene_[ID].json` file.
    -   Models will be stored in a dedicated `models/` directory.
-   **Scene Data Structure (`scene.model.ts`):**
    ```typescript
    export interface SceneObject {
      objectId: string; // Unique ID for this object instance
      modelId: string;  // ID of the model from the ModelManager
      position: { x: number; y: number; z: number };
      rotation: { x: number; y: number; z: number };
      scale: { x: number; y: number; z: number };
    }

    export interface Scene {
      id: string;
      name: string;
      lastModified: number;
      objects: SceneObject[];
      backgroundImageUrl?: string; // Optional background from the asset API
    }
    ```

## 7. Native Integration with Capacitor

-   **Filesystem:** The `@capacitor/filesystem` plugin is critical. It will be used by the `StorageService` to:
    -   Read and write JSON files for scene data and manifests.
    -   Save downloaded 3D model files (e.g., `.glb`).
    -   Check for the existence of files and directories.
-   **Directory Structure on Device:** The `StorageService` will ensure the following structure is maintained within the app's data directory (e.g., `FilesystemDirectory.Data`):
    ```
    /
    |-- scenes/
    |   |-- scene_abc123.json
    |   |-- scene_def456.json
    |-- models/
    |   |-- model_xyz789.glb
    |-- manifest.json
    ```

## 8. User Workflow Example: Creating a Scene

1.  **Open App:** User opens the app and sees the `SceneListPage` (or the last opened scene). The side menu allows navigation to "Scenes" and "Model Library".
2.  **Add a Model:** User navigates to the "Model Library" page. The page is initially empty.
    -   User clicks an "Import Model" button, which opens the native file picker.
    -   User selects a `.glb` model from their device's storage.
    -   The `ModelManagerService` is called. It uses `StorageService` to copy the file into the app's internal data directory as `models/some_unique_id.glb`.
    -   An entry is added to `manifest.json` mapping the model's name to its new local file path.
    -   The UI updates to show the new model in the library.
3.  **Create a Scene:** User navigates back to the "Scenes" page and clicks "New Scene".
    -   `SceneManagerService` creates a new `Scene` object with a unique ID, saves it as `scenes/scene_[ID].json`, and updates the manifest.
    -   The app navigates to the `editor/[ID]` route.
4.  **Edit the Scene:** The `EditorPage` loads.
    -   It shows an empty 3D world via the `<app-three-viewport>` component.
    -   User clicks an "Add Model" button. A modal appears showing models from the `ModelManagerService`.
    -   User selects "Low-Poly Tree".
    -   The `EditorPage` calls `ThreeJsService.addObject('path/to/model')`.
    -   `ThreeJsService` loads the model and adds it to the center of the scene.
5.  **Manipulate Object:**
    -   User taps on the tree in the viewport. `ThreeJsService` uses raycasting to detect the selection and emits the selected object through its `selectedObject$` observable.
    -   The `PropertiesPanelComponent`, subscribed to this observable, now displays the tree's position, rotation, and scale.
    -   User changes the X position in the input field. The component calls `ThreeJsService.setSelectedObjectPosition(...)`. The tree moves in the viewport.
6.  **Save Scene:** User clicks "Save".
    -   The `EditorPage` gets the current scene state from `ThreeJsService` and calls `SceneManagerService.saveScene(currentSceneData)`.
    -   The service overwrites the `scene_[ID].json` file with the updated data.

## 9. API Integration: `AssetService`

-   The `AssetService` will be responsible for fetching data from the placeholder image API (`https://picsum.photos/`).
-   **Usage Example:** In the `EditorPage`, there could be a button "Set Background".
    -   This button's click handler would call `AssetService.getRandomImage()`.
    -   The service makes an HTTP request.
    -   Upon receiving the image URL, it could be set as the `backgroundImageUrl` property in the current `Scene` object.
    -   `ThreeJsService` would then be instructed to update the scene's background using the new image URL, loading it as a texture.
-   This provides a simple, clear use case for integrating an external API as requested.
